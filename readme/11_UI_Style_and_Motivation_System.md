# 11_UI_Style_and_Motivation_System

## 1. 设计目标

本项目的 UI 不追求炫技，而追求三件事：

1. 统一风格，形成一致的视觉语言
2. 漂亮但不过度华丽，保持工具感与专业感
3. 触发学习欲望、答题欲望与持续反馈欲望

本项目的 UI 应该呈现一种“轻知识工作台”的感觉：
- 不是娱乐平台
- 不是繁重后台系统
- 不是学术论文网站
- 而是一个让人愿意持续刷题、持续复盘、持续进步的学习空间

---

## 2. 视觉定位

### 2.1 关键词

- 安静
- 清晰
- 专注
- 轻盈
- 温和的专业感
- 学习激励
- 结构化信息
- 轻量而有层次

### 2.2 视觉风格

- 整体采用浅色主题为主
- 背景保持柔和的灰白或冷白，不使用高饱和大面积纯色
- 主色建议使用稳重的蓝紫系或蓝绿色系
- 强调色用于按钮、进度、状态提示，而非大面积填充
- 卡片、分区、徽标、进度条承担视觉层次
- 保持适当留白，避免信息压迫

### 2.3 禁止的风格

- 过度花哨的渐变大背景
- 过强的霓虹效果
- 大面积高饱和色块
- 过多动效打扰学习
- 容易造成疲劳的深色压迫式界面

---

## 3. 统一设计语言

### 3.1 设计一致性原则

全站必须统一以下视觉规则：

- 圆角体系一致
- 阴影体系一致
- 卡片边框一致
- 按钮层级一致
- 字体层级一致
- 间距体系一致
- 徽标颜色语义一致
- 状态提示样式一致

### 3.2 建议的视觉组件体系

建议将 UI 统一成以下视觉组件：

- 页面标题区 `PageHeader`
- 概览信息区 `SummaryCards`
- 功能入口区 `FeatureGrid`
- 数据列表区 `DataTable / ListCards`
- 详情解释区 `DetailPanel`
- 学习进度区 `ProgressPanel`
- 任务状态区 `TaskStatusPanel`
- 空状态区 `EmptyState`
- 错误状态区 `ErrorState`
- 加载骨架区 `SkeletonState`

### 3.3 字体与排版

- 标题要更有层级感
- 正文保持可读性优先
- 列表项信息尽量精炼
- 讲解类内容使用舒适行高
- 重要信息采用更大字重，而不是更花哨颜色

---

## 4. 激发学习欲望的 UI 策略

### 4.1 反馈即时化

用户一旦完成某个动作，界面要立刻给出反馈：

- 导入成功
- 简历解析成功
- 题目生成成功
- 评分完成
- 复习完成
- 掌握度提升

反馈要明确，但不要夸张。

### 4.2 进度可视化

学习系统必须让用户直观看到“自己正在进步”：

- 总练习次数
- 当前掌握率
- 当前待复习数量
- 最近错题数量
- 简历相关题目完成率
- 技术栈薄弱分布

### 4.3 目标感设计

前端应帮助用户形成“今天要完成什么”的感觉：

- 今日待练习
- 今日待复习
- 本周薄弱点
- 本次面试目标
- 当前简历重点技术栈

### 4.4 连续性设计

界面应鼓励用户继续下一步，而不是一次任务结束后没有下文：

- 导入后直接引导出题
- 题目详情后直接引导模拟面试
- 模拟面试后直接引导复盘
- 复盘后直接引导复习

---

## 5. 激发答题欲望的 UI 策略

### 5.1 题目要“像任务”而不是“像文本”

题目卡片要具备“想点开”的吸引力：
- 标题清晰
- 难度明确
- 来源明确
- 标签明确
- 有可预期的奖励感

### 5.2 题目详情要有层级

题目详情不应一屏堆满内容，而应分层呈现：
- 原题
- 来源
- 相关知识
- 简短提示
- 讲解
- 追问
- 复盘

### 5.3 答题过程要有节奏感

- 先看到题目
- 再输入答案
- 再看到评分
- 再看到缺失点
- 再看到讲解
- 再进入下一题

这个节奏要让用户愿意继续答，而不是答完就退出。

### 5.4 分数与进步感

评分不要只是“通过/不通过”，而应突出：
- 哪些点做对了
- 哪些点还缺失
- 距离“熟练”还差多少
- 下一次如何提升

---

## 6. 颜色与语义系统

### 6.1 建议色彩角色

- 主色：用于核心按钮、选中态、进度条
- 成功色：用于已完成、掌握、通过
- 警示色：用于待复习、未通过、注意项
- 信息色：用于来源、标签、知识节点
- 中性色：用于背景、边框、说明文字

### 6.2 颜色语义统一

- 绿色：掌握、通过、完成
- 黄色：待复习、注意、部分正确
- 红色：错误、失败、严重薄弱
- 蓝色：信息、引导、知识
- 紫色：AI 生成、讲解、推荐

### 6.3 不同页面要遵守同一语义

不能出现同样的状态在不同页面颜色表达不同的问题。

---

## 7. 交互层级与动效原则

### 7.1 动效原则

- 动效只服务于状态切换，不服务于炫技
- 进入、展开、切换、提交时可有轻量过渡
- 不要长时间、强烈、复杂的动效

### 7.2 交互层级

- 一级动作：主要 CTA
- 二级动作：切换、筛选、展开
- 三级动作：查看详情、查看来源、查看讲解

### 7.3 学习过程中的微反馈

建议增加轻量微反馈：
- 悬停态
- 按下态
- 选中态
- 加载态
- 成功态
- 失败态

这些微反馈会显著提升“系统活着”的感觉。

---

## 8. 页面级 UI 方向建议

### 8.1 首页

首页应该像一个“学习驾驶舱”：
- 上方展示概览
- 中间展示功能入口
- 下方展示最近任务与学习进度
- 强化“继续学习”的主按钮

### 8.2 导入页

导入页应该像一个“知识入口站”：
- 明确区分题目导入与简历导入
- 给出结果预览
- 给出下一步去向

### 8.3 题目页

题目页应该像一个“题库工作台”：
- 搜索、筛选、分页清晰
- 题卡有层次感
- 来源和难度一眼可见

### 8.4 题目详情页

题目详情页应该像一个“学习讲义页”：
- 信息分区明确
- 证据链路可视化
- 讲解和追问分开

### 8.5 模拟面试页

模拟面试页应该像一个“面试舱”：
- 题目集中
- 输入区干净
- 历史对话清晰
- 评分和建议反馈明显

### 8.6 统计页

统计页应该像一个“进步仪表盘”：
- 指标聚合优先
- 图表克制但清楚
- 强调趋势和改善，而不是仅仅展示数字

---

## 9. 组件设计要求

### 9.1 可复用优先

UI 组件必须可复用，不要每个页面重新写一套样式。

### 9.2 语义化组件优先

优先抽象语义组件，而不是纯样式组件，例如：
- `QuestionBadge`
- `ResumeBadge`
- `DifficultyBadge`
- `MasteryBadge`
- `AiGeneratedBadge`
- `SourceBadge`

### 9.3 抽象层级建议

- 原子组件：按钮、输入框、徽标、卡片
- 业务组件：题目卡、简历卡、评分卡、复习卡
- 页面组件：首页、导入页、面试页、统计页

---

## 10. 落地建议

### 10.1 先统一全局风格，再做页面细节

建议优先改：
- `layout.tsx`
- `globals.css`
- 全局导航
- 全局卡片/按钮/徽标样式

### 10.2 再逐页升级

优先升级：
- 首页
- 导入页
- 题目列表页
- 模拟面试页
- 统计页

### 10.3 最后补视觉激励点

最后再加入：
- 学习进度动画
- 完成反馈
- 连续答题激励
- 技术栈薄弱提示
- 继续下一题引导

---

## 11. 给 Claude Code 的执行原则

当 Claude Code 依据本文件改 UI 时，应遵守：

- 先统一全局设计语言
- 再统一组件样式
- 再逐页增强层次感
- 不要过度装饰
- 不要牺牲可读性
- 不要让 UI 与业务逻辑耦合
- 不要让页面看起来像零散拼接

本项目的 UI 目标是让人愿意长时间学习，愿意持续刷题，愿意一题接一题地往下做。
