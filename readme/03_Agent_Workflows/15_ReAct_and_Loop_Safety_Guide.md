# 15_ReAct_and_Loop_Safety_Guide

## 1. 目标

本文件专门说明本项目在使用 LangGraph ReAct / 循环型 Agent 时，必须遵守的安全边界与成本控制策略。

ReAct 的强项是“思考 → 工具调用 → 观察 → 再思考”，但如果没有循环上限，就可能出现：
- 无限追问
- 无限检索
- 无限重试
- 无限工具调用
- 模型费用持续累积
- 用户长时间等待却得不到收敛结果

因此，ReAct 不是“越多循环越好”，而是“必须可控地循环”。

---

## 2. 核心原则

### 2.1 必须有上限

所有循环型 Agent 都必须显式设置上限，例如：
- `max_turns`
- `max_iterations`
- `max_steps`
- `max_tool_calls`

上限必须是显式配置，而不是依赖默认行为。

### 2.2 必须有退出条件

循环不仅要有次数上限，还必须有明确的退出条件，例如：
- 用户回答已达到及格标准
- 检索结果不足以继续推进
- 工具返回为空或连续失败
- 已达到深挖目标
- 已完成预设的追问轮次

### 2.3 必须有收敛结果

达到上限后，系统不能继续空转，必须输出：
- 最终总结
- 当前阶段性结论
- 后续建议
- 可进入复习或下一题的提示

### 2.4 必须控制资源消耗

循环型 Agent 应尽量避免：
- 对同一工具重复调用
- 对同一上下文反复检索
- 对同一答案重复评分
- 无意义的“再想一想”重试

---

## 3. 适合使用 ReAct 的场景

ReAct 只适合以下“需要动态决策”的任务：
- 简历驱动模拟面试
- 动态追问
- 动态出题
- 基于用户回答的分支式面试官决策
- 基于检索结果选择下一步动作

不适合的场景：
- 分页
- CRUD
- 单次结构化抽取
- 单次分类
- 普通讲解生成
- 纯统计聚合

---

## 4. 推荐的循环控制设计

### 4.1 面试官 Agent

建议设置：
- `max_turns = 5` 或 `6`
- 当用户答得足够好时可提前结束
- 当同一主题连续追问无收益时必须切题或结束

### 4.2 出题 Agent

建议设置：
- `max_iterations = 3` 到 `5`
- 每次生成后检查是否重复、是否过难、是否偏题
- 达到上限后直接输出题目结果，不再继续优化

### 4.3 检索增强 Agent

建议设置：
- `max_tool_calls` 控制检索次数
- 如果连续几轮检索没有新增信息，就停止循环

---

## 5. 实现层要求

### 5.1 LangGraph 路由必须有收敛边

不能只有循环边，没有结束边。
每个有循环的 Graph 都必须能明确走向 END。

### 5.2 状态中必须保存轮次

建议在 state 或 metadata 中保存：
- 当前轮次
- 最大轮次
- 已调用工具次数
- 上一次工具结果
- 是否已经达到收敛条件

### 5.3 需要在服务层兜底

即使 Graph 路由设计有误，服务层也要再次校验：
- 是否超过上限
- 是否需要强制结束
- 是否应该返回保守结果

### 5.4 前端必须能看见“已收敛”

前端不应该只显示“完成”，还要能看到：
- 本次共进行了几轮
- 是否达到最大轮次
- 是否因为收敛条件提前结束
- 是否存在未完全展开的后续建议

---

## 6. 给 Claude Code 的执行原则

当 Claude Code 生成或修改 ReAct / 循环型 Agent 时，必须先确认：
- 是否设置了最大轮次或最大步数
- 是否存在明确退出条件
- 是否存在强制收敛路径
- 是否避免了无限工具调用
- 是否保留了成本控制兜底

如果没有这些约束，就不允许把它实现成真正的 ReAct 循环。

---

## 7. 最终结论

ReAct 是本项目中非常重要的能力，但它必须是“有限循环、可收敛、可解释、可控成本”的。

任何没有上限的循环 Agent，都会给系统带来资源消耗、等待时间和费用风险，因此必须在架构文档和 MVP 文档中明确禁止。
