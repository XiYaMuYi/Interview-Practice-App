# 16_Resume_Workflow_Refactor_Spec

## 1. 目标

本文件用于指导后续代码重构，把“简历生成面试题”和“简历驱动面试”从当前较混杂的实现，重构为两条清晰、可维护、可扩展的业务链路。

本项目现状中，这两类能力都依赖简历输入、题目抽取、知识节点映射、LLM 调用与数据库入库，但它们的目的并不相同：

- **简历生成面试题**：把简历或长文本加工成题库资产
- **简历驱动面试**：把简历变成一个可多轮互动的面试官 Agent

重构目标是把这两条链路在代码层、文档层、接口层彻底区分清楚，同时保留它们共享的底层基础设施。

---

## 2. 现状判断

### 2.1 当前代码里的主要入口

从当前后端代码看，相关能力主要分布在：

- `backend/app/services/import_service.py`
  - 负责文本导入、切块、题目抽取、知识节点抽取、入库
- `backend/app/services/ai_service.py`
  - 负责讲解、评分、追问、模拟面试、复盘总结
- `backend/app/infra/llm/prompt_registry.py`
  - 负责 prompt 集中管理
- `backend/app/infra/llm/gateway.py`
  - 负责统一模型调用
- `backend/app/graphs/*`
  - 负责 LangGraph 工作流编排
- `backend/app/api/v1/routes/*`
  - 负责 HTTP 接口暴露

### 2.2 当前的混杂点

当前实现里，容易混在一起的地方包括：

- 简历解析、题目抽取、知识点抽取、题库落库放在同一条导入链路里
- 面试官 Agent 的追问、评分、总结与讲解逻辑在 `ai_service.py` 中交叉存在
- 一些 prompt 虽然已经集中到了 `PromptRegistry`，但对应功能边界还不够清楚
- 前端和后端对“简历生成面试题”和“简历驱动面试”的理解容易混淆

## 3. 两条链路的职责边界

### 3.1 简历生成面试题

这条链路是“内容生产管道”。它的职责是把简历、笔记、题库原文等内容，转换成结构化题目资产。

#### 目标输出
- 题目列表
- 题目分类
- 难度等级
- 领域标签
- 知识节点
- 参考答案或答案摘要

#### 适用场景
- 长简历导入
- 题库批量导入
- 非结构化笔记整理成题库
- 题目资产沉淀

#### 主要特征
- 偏批处理
- 偏静态结果
- 偏落库
- 可流式，但最终目标是产出题库数据

### 3.2 简历驱动面试

这条链路是“交互式面试官 Agent”。它的职责是根据简历、题目和用户回答，动态决定下一步问什么。

#### 目标输出
- 首题
- 追问
- 评价
- 继续问还是结束
- 最终总结

#### 适用场景
- 模拟面试
- 动态追问
- 回答评分
- 面试复盘

#### 主要特征
- 偏多轮交互
- 偏动态决策
- 偏上下文连续
- 需要明确轮次上限和收敛条件

## 4. 重构后的推荐架构

### 4.1 内容生产管道

建议命名为：`resume_to_question_pipeline`

职责：
- 读取简历或导入文本
- 切块
- 语义解析
- 抽取知识节点
- 抽取题目
- 分类与打标
- 入库
- 流式输出进度

#### 建议代码边界
- `import_service.py`
  - 只负责导入与抽题，不负责面试交互
- `question_service.py`
  - 负责题库结构化与检索
- `prompt_registry.py`
  - 负责抽题、分类、知识点抽取 prompt

### 4.2 面试交互管道

建议命名为：`resume_interview_agent`

职责：
- 读取简历上下文
- 检索知识点与题库
- 生成首题
- 根据用户回答进行评分
- 生成追问
- 判断是否继续或结束
- 输出最终总结

#### 建议代码边界
- `ai_service.py`
  - 负责面试逻辑编排
- `interview_graph.py`
  - 负责 Agent 状态机与分支路由
- `interview_nodes.py`
  - 负责具体节点动作

## 5. 建议的代码改造点

### 5.1 `backend/app/services/import_service.py`

#### 现状
当前它承担了导入、切块、题目抽取、知识点抽取、保存等职责。

#### 重构要求
- 保留“内容导入 + 抽题”功能
- 明确该服务只负责内容生产，不负责面试交互
- 将题目抽取、知识节点抽取、保存结果拆为可复用内部方法
- 为流式导入保留任务状态和中间结果
- 不要让它混入面试追问逻辑

### 5.2 `backend/app/services/ai_service.py`

#### 现状
当前它同时承载讲解、评分、追问、模拟面试、复盘等多类逻辑。

#### 重构要求
- 将“讲解”与“面试交互”边界明确区分
- 模拟面试相关方法应统一归入交互式 Agent 逻辑
- 评分、追问、结束条件、总结应构成一条完整闭环
- 不要让它承担题库导入职责

### 5.3 `backend/app/graphs/interview_graph.py`

#### 重构要求
- 如果使用 ReAct 场景，必须限定在“简历驱动面试官”这条链路里
- 明确循环上限、退出条件与收敛结果
- 不要把所有 Agent 场景都做成同一种循环图

### 5.4 `backend/app/infra/llm/prompt_registry.py`

#### 重构要求
- 抽题 prompt、分类 prompt、追问 prompt、评分 prompt、复盘 prompt 必须分开管理
- 每个 prompt 都要明确用途、输入、输出、版本
- prompt 不要功能错位

## 6. 功能重构的目标状态

### 6.1 简历生成面试题链路最终应呈现为

```text
简历 / 长文本
    ↓
切块
    ↓
语义解析
    ↓
知识节点抽取
    ↓
题目抽取
    ↓
分类 / 打标 / 难度评估
    ↓
确认 / 编辑
    ↓
入库到题库与知识库
```

### 6.2 简历驱动面试链路最终应呈现为

```text
简历 / 题目
    ↓
检索相关知识点与题库
    ↓
生成首题
    ↓
用户回答
    ↓
评分 / 追问 / 讲解
    ↓
判断是否继续
    ↓
最终总结 / 复盘
```

## 7. 前端配合要求

前端也应区分两个入口：

### 7.1 题库生成入口
- 文件上传
- 解析进度
- 题目候选列表
- 人工确认后入库

### 7.2 面试入口
- 选择简历或题目开始面试
- 看到当前轮次
- 看到系统追问
- 看到评分与总结
- 能回看面试过程

---

## 8. 给 Claude Code 的执行要求

当 Claude Code 按本文件执行时，必须遵守：

1. 先分清“内容生产”和“面试交互”两条链路
2. 不要把这两类功能混写到同一个 service 里
3. 遇到混杂逻辑时，优先拆分职责，而不是继续堆代码
4. 所有长任务必须支持流式进度与增量保存
5. 所有循环型 Agent 必须设置上限与收敛路径
6. 所有 prompt 必须和功能一一对应，不能错位

## 10. 最终结论

如果要让系统后续可持续扩展，就必须把：

- 简历生成面试题
- 简历驱动面试

拆成两条清晰的业务链路。

一条负责“生成题目资产”，一条负责“进行动态面试”。
两者共享底座，但不能混成一锅。
