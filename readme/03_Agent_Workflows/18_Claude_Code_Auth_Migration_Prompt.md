## 二、任务目标

请把现有项目从“已有登录功能但未真正接入鉴权”的状态，升级为以下能力：

### 1. 登录与访问控制
- 所有需要登录的页面都必须基于统一鉴权中间件或后端校验进行保护。
- 未登录用户只能访问公开页面。
- 已登录但未审核用户只能访问受限页面或审核提示页。
- 已审核用户可以进入个人工作台。
- 管理员可进入管理员审核后台。

### 2. 数据隔离
- 公共题库保持共享。
- 用户上传的简历、资料、题目、学习记录、历史记录、对话记录、错题集等全部按用户隔离。
- 所有私有接口都必须带 `user_id` 级别过滤。

### 3. 管理员审核注册
- 注册成功后默认进入待审核状态。
- 管理员可在后台查看待审核用户。
- 管理员可批准或拒绝用户。
- 未审批通过的用户不能正常登录主系统。

### 4. 页面与路由
- 增加登录页、注册页、审核中提示页。
- 增加管理员审核后台入口。
- 个人工作台只展示当前用户自己的内容。
- 保留公共题库的全局可见能力。

---

## 三、实施原则

1. 先后端，后前端。
2. 先认证与授权，再业务页面。
3. 先打通最小闭环，再优化体验。
4. 不要只做前端隐藏按钮，必须做后端校验。
5. 所有私有资源必须强制绑定用户身份。
6. 管理员权限必须显式判断，不能默认透传。

---

## 四、推荐开发步骤

### 第 1 步：梳理现有鉴权与登录链路
- 找出当前登录实现位置。
- 找出当前用户上下文是如何传递的。
- 找出所有需要登录保护的路由与接口。
- 找出所有应当变成“用户私有”的数据模型与接口。

### 第 2 步：设计统一的用户状态模型
- 补充角色字段 `role`。
- 补充审核状态字段 `status`。
- 约定状态机：`pending / approved / rejected / disabled`。
- 约定角色：`user / admin`。

### 第 3 步：实现后端认证与授权
- 提供统一的当前用户解析。
- 在后端对每个私有接口进行登录校验。
- 对管理员接口增加管理员校验。
- 对未审核用户进行访问限制。

### 第 4 步：实现注册审核流程
- 注册接口创建待审核用户。
- 管理员后台可查看待审核列表。
- 管理员审批后更新用户状态。
- 审核状态影响后续登录结果。

### 第 5 步：重构私有数据归属
- 所有简历、导入、题目、笔记、学习记录等表补齐 `user_id`。
- 查询接口统一按当前用户过滤。
- 新建数据时自动写入用户归属。
- 防止越权读取与修改。

### 第 6 步：重构前端页面与路由
- 登录页与注册页接入真实鉴权。
- 个人工作台按用户展示数据。
- 添加“审核中”状态页。
- 管理员后台单独路由与导航入口。

### 第 7 步：补充保护与提示体验
- 未登录跳转登录页。
- 待审核显示明确提示。
- 无权限显示友好错误。
- 对关键操作增加加载与错误反馈。

## 七、验收标准

当任务完成时，应满足：

- 游客无法进入私有工作台。
- 注册用户默认不能直接使用主系统，必须审核通过。
- 普通用户只能看到自己的简历、导入记录、题目、学习过程、历史记录。
- 公共题库仍然对所有用户可见。
- 管理员可以审核注册用户。
- 管理员入口与普通用户入口分离。
- 后端对私有接口有真实权限控制，不依赖前端隐藏。

---

## 八、建议的执行策略

如果任务范围较大，请不要一次性改所有功能。建议按以下顺序推进：

1. 先把认证状态跑通。
2. 再把审核状态接入登录。
3. 再把私有数据按用户隔离。
4. 再做管理员审核后台。
5. 最后统一修前端交互和权限提示。

- 不要遗漏任何私有查询接口。
- 管理员权限与普通用户权限不要混用。
