# 阅读说明

> 本文档用于指导你或 Claude Code 按正确顺序阅读后端 Agent / RAG 代码，快速建立系统心智模型。
>
> 目标不是逐行背代码，而是先理解“状态如何流转、节点如何编排、服务如何承接、检索如何工作、接口如何对外暴露”。

---

## 1. 这份文档的用途

本项目的后端并不是一个简单的 CRUD 服务，而是由以下几层共同组成：

- **LangGraph Agent 工作流层**：负责问答、讲解、追问、评分、复习等多步智能流程
- **AI 业务编排层**：负责把模型调用封装成稳定业务能力
- **RAG 检索层**：负责向量存储与相似性检索
- **题库与学习服务层**：负责题目、标签、学习记录、知识节点等核心业务
- **API 层**：负责对前端暴露稳定接口

如果你一开始就直接看路由或数据库实现，很容易只看到局部；如果先看状态和图，再看节点和服务，就能很快串起整个系统。

---

## 2. 推荐阅读顺序

建议按下面顺序阅读。

### 第 1 步：先看状态定义

#### 1. `backend/app/graphs/states.py`

这是整个 Agent / RAG 工作流的**状态契约**。

你要先看它，因为：
- 所有 LangGraph 节点都围绕这个状态读写
- 它决定了输入、理解、生成、决策、持久化字段有哪些
- 你后面看任何 graph 或 node，都会反复看到这些字段

重点关注：
- `input_text`
- `question_text`
- `question_type`
- `domain_type`
- `difficulty_level`
- `tags`
- `knowledge_points`
- `retrieval_hits`
- `answer_short`
- `answer_detail`
- `user_answer`
- `user_score`
- `mastery_level`
- `next_action`
- `persist_flag`
- `metadata`

你可以把它理解为：**一条工作流运行时的“共享记忆体”**。

---

### 第 2 步：看最简单的图

#### 2. `backend/app/graphs/explanation_graph.py`

这是最简单的线性工作流。

流程是：

- `Extractor -> Classifier -> Explainer -> Persister`

为什么先看它：
- 它最直观
- 没有复杂分支
- 很适合建立“节点如何串起来”的基础认知

你应该重点理解：
- 图是怎么创建的
- 节点是怎么注册的
- 线性边是怎么连接的
- 为什么最后要走 `Persister`

看完这一个，你就能理解 LangGraph 最基础的结构。

---

### 第 3 步：看主交互图

#### 3. `backend/app/graphs/interview_graph.py`

这是项目里最重要的 Agent 图之一。

它是面试模拟的主工作流，包含：

- 题目抽取
- 分类
- 检索
- 讲解
- 追问
- 评价
- 持久化

流程大致是：

- `Extractor -> Classifier -> Retriever -> Explainer/Interviewer -> Evaluator -> Persister`

你要重点看：
- `route_after_extractor`
- `route_after_classifier`
- `route_after_retriever`
- `route_after_interviewer`
- `route_after_evaluator`
- `route_after_explainer`

这些 routing 函数决定了流程如何分支。

你要形成的理解是：
- **Graph 负责流程控制**
- **Node 负责具体动作**
- **State 负责在流程间传递数据**

这个文件是整个 Agent 架构的关键入口。

---

### 第 4 步：看复习图

#### 4. `backend/app/graphs/review_graph.py`

这是间隔复习 / SM-2 相关的工作流。

它的重点不是“问答”，而是：
- 用户刚才的表现如何
- 复习间隔如何变化
- 掌握度如何调整
- 下一次什么时候复习

流程是：

- `LoadQuestion -> Evaluate -> ScheduleReview -> Persist`

你要重点看：
- `WRONG_THRESHOLD`
- `INITIAL_INTERVAL_DAYS`
- `review_evaluator_node`
- `schedule_review_node`

这个图帮助你理解：
**学习系统不是只做一次答题，而是要形成复习闭环。**

---

### 第 5 步：看节点实现

#### 5. `backend/app/graphs/nodes/interview_nodes.py`

这是最核心的节点实现文件。

里面包含：
- `extractor_node`
- `classifier_node`
- `retriever_node`
- `explainer_node`
- `interviewer_node`
- `evaluator_node`
- `persister_node`

你可以把它理解成：
**图上的每一个方块具体怎么做事。**

阅读时建议逐个看：

##### `extractor_node`
作用：从原始输入中提取出题目文本。

要关注：
- 它如何处理空输入
- 它如何调用 LLM 做抽取
- 它为什么要返回 `run_id`

##### `classifier_node`
作用：给题目分类、分领域、估难度、打标签。

要关注：
- `question_type`
- `domain_type`
- `difficulty_level`
- `tags`
- `knowledge_points`
- `prerequisites`

##### `retriever_node`
作用：检索相关题目和知识上下文。

要关注：
- 现在为什么还是 MVP 占位
- 未来如何接 pgvector 和知识检索

##### `explainer_node`
作用：生成分层讲解。

要关注：
- 如何输出 `answer_short`
- 如何输出 `answer_detail`
- 如何输出 `common_pitfalls`

##### `interviewer_node`
作用：扮演面试官追问。

要关注：
- 如何利用 `chat_history`
- 如何增加追问轮次
- 如何返回 `followup_questions`

##### `evaluator_node`
作用：评估用户回答。

要关注：
- 如何生成分数
- 如何提炼 `feedback`
- 如何生成 `mastery_level`
- 如何决定是否需要复习

##### `persister_node`
作用：告诉服务层“当前工作流结果可以落库了”。

要关注：
- 它本身不写数据库
- 它只是发出保存信号
- 真实保存应该回到 service/repository 层

这个文件是理解“Agent 如何真正做事”的核心。

---

### 第 6 步：看 Agent 业务编排层

#### 6. `backend/app/services/ai_service.py`

这是 AI 能力的业务入口。

里面主要有：
- `explain_question`
- `start_interview`
- `evaluate_answer`
- `generate_followup`
- `handle_interview_turn`

这个文件的作用是：

- 把模型调用包装成业务能力
- 给前端或图层提供稳定接口
- 统一处理解释、模拟面试、评分、追问等功能

你要重点理解：
- 这里不是直接散落 prompt
- 它是业务编排层
- 它通过 `llm_gateway` 统一调用模型
- 它会把结果写回题目或返回给上层流程

可以把它理解成：
**“对话与讲解的业务总管”**。

---

### 第 7 步：看题库与检索业务层

#### 7. `backend/app/services/question_service.py`

这是题库业务的核心服务。

你要重点关注它在做什么：
- 创建题目
- 去重
- 分类
- 搜索
- 语义检索
- 关联知识节点
- 处理题目上下文

它的角色是：
**题库和 RAG 的业务承接层。**

阅读时建议重点理解：
- 为什么要用 `content_hash` 去重
- 为什么所有来源最终都要收敛到 canonical question record
- 为什么语义搜索要走 embedding 和向量库，而不是简单文本匹配

---

### 第 8 步：看向量检索基础设施

#### 8. `backend/app/infra/vectorstore/pgvector_store.py`

这是最底层的向量检索实现。

它负责：
- 存储 embedding
- 查询相似题
- 删除题目向量

你要重点理解：
- 它为什么“很薄”
- 为什么高层业务规则不应该写在这里
- 为什么它只做向量存取和相似度搜索

可以把它理解为：
**RAG 的地基。**

---

### 第 9 步：看 LLM 网关和 Prompt 管理

后续建议继续看：

#### 9. `backend/app/infra/llm/gateway.py`

职责：
- 所有模型调用的统一入口
- 负责把业务请求转成 provider 调用
- 处理聊天、JSON 输出、重试、容错等

#### 10. `backend/app/infra/llm/prompt_registry.py`

职责：
- 集中管理 prompt
- 管理 prompt 版本
- 避免 prompt 分散在各个函数里

这两个文件是后面稳定迭代的关键。

---

### 第 10 步：看 API 层

#### 11. `backend/app/api/v1/routes/*.py`

建议按功能看：
- `question_routes.py`
- `study_routes.py`
- `chat_routes.py`
- `ai_routes.py`
- `resume_routes.py`
- `import_routes.py`

职责：
- 把前端请求转给 service
- 返回稳定 DTO
- 不写业务逻辑
- 不直接操作模型
- 不直接拼复杂流程

API 层只是“门面”，真正的逻辑在 service / graph / infra。

---

### 第 11 步：看仓储层

#### 12. `backend/app/infra/repositories/*`

职责：
- 封装数据库读写
- 避免 service 层直接写 SQL
- 统一 CRUD 和查询接口

理解仓储层后，你就能知道：
- 哪些逻辑应该在 service
- 哪些逻辑应该在 repository
- 哪些逻辑应该下沉到 infra

## 3. 你应该形成的整体心智模型

看完这套代码后，你要形成下面这几个认识：

### 3.1 Agent 不是一个函数，而是一套状态机

流程是分步的：
- 输入
- 抽取
- 分类
- 检索
- 讲解
- 追问
- 评价
- 保存

每一步都可能单独变化。

---

### 3.2 State 是整个系统的“共享上下文”

如果你要理解某一步为什么这么做，先去看它读写了哪些 state 字段。

---

### 3.3 Graph 管流程，Node 管动作，Service 管业务，Infra 管基础设施

这是本项目最重要的分层原则。

- **Graph**：决定流转方向
- **Node**：决定单个步骤做什么
- **Service**：决定业务规则和组合方式
- **Infra**：决定如何与数据库、模型、向量库、文件系统交互

---

### 3.4 RAG 不只是检索，而是“检索 + 组织 + 生成 + 评价”

题库系统不是单纯搜相似题，而是：
- 先找知识
- 再组织答案
- 再生成讲解
- 再评价回答
- 再沉淀学习数据

---

## 6. 结语

这个项目不是简单的题库系统，而是一个面向面试训练的 AI 学习系统。

所以阅读代码时，不要只看“某个函数做了什么”，而是要看：
- 它在整个工作流里处于什么位置
- 它和上下游模块如何协作
- 它为什么被拆成这样
- 它如何支撑未来继续迭代

只要你先理解了状态、图、节点、服务、基础设施这五层，后面再看具体实现就会很快上手。
