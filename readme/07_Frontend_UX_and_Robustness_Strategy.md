# 07_Frontend_UX_and_Robustness_Strategy

## 1. 背景与目标

本项目当前前端已经具备较完整的基础页面：

- 首页 `web/src/app/page.tsx`
- 导入页 `web/src/app/import/page.tsx`
- 题目列表页 `web/src/app/questions/page.tsx`
- 题目详情页 `web/src/app/questions/[id]/page.tsx`
- 学习页 `web/src/app/study/page.tsx`
- 模拟面试页 `web/src/app/interview/page.tsx`
- 统计页 `web/src/app/stats/page.tsx`

从现有实现看，当前前端已经完成了“可用”的最小闭环，但存在以下明显特征：

- 多个页面仍然使用大范围拉取数据的方式
- 页面状态处理较为分散，加载态、错误态、空态不统一
- AI 生成内容与确定性数据的区分不够显式
- 导入、练习、统计、面试四条链路之间的联动还不够强
- 简历驱动面试这一新增核心能力需要与现有页面统一入口和统一交互范式衔接

因此，本文件的目标不是重做前端，而是基于现有页面结构，补齐一套更健壮、更可持续迭代的前端交互与数据加载策略，让 Claude Code 后续可以在现有基础上做增量开发。

## 2. 现有前端状态分析

### 2.1 已有优势

#### 首页
- 已有功能入口卡片，适合作为统一导航中心
- 已展示最近导入与基础统计，具备做“最近任务 / 最近简历 / 最近练习”的扩展基础

#### 导入页
- 已支持文本导入与文件上传
- 已有拖拽上传与状态反馈
- 非常适合扩展为“题目导入 + 简历导入”的双入口页面

#### 题目列表页
- 已有搜索与筛选雏形
- 已有题目卡片展示
- 很适合作为分页与来源筛选的核心页

#### 题目详情页
- 从学习页逻辑可见，后端已支持题目详情、标签、知识节点、讲解等结构
- 适合展示来源链路、知识映射、AI 讲解版本等信息

#### 学习页
- 已集成练习、复习、统计三个 tab
- 已有提交回答、评分、讲解、复习列表等核心流程
- 是未来做“简历驱动训练模式”的最佳承载页

#### 模拟面试页
- 已有完整的面试开始、追问、评分、总结流程
- 适合作为“简历模式”的主战场，而不是单独新增一套页面

#### 统计页
- 已有学习概览和最近记录
- 适合引入分页、筛选和来源维度统计

### 2.2 当前问题

#### 问题 1：数据读取偏全量
当前多个页面在加载时直接请求较大的数据集，例如：
- `questions` 页一次取较多题目
- `study` 页一次取较多题目用于练习
- `stats` 页一次取较多学习记录

当题目量、学习记录量、简历解析量增长后，这种方式会导致：
- 首屏慢
- 请求负载高
- 前端渲染压力大
- 搜索/筛选不够精准

#### 问题 2：状态处理不统一
现有页面各自实现 loading / error / empty state，风格和行为不完全统一。

#### 问题 3：AI 结果与源数据没有足够强的视觉区分
例如：
- 哪些内容来自简历解析
- 哪些内容是 AI 生成
- 哪些内容需要人工确认
- 哪些内容已确认可直接进入题库

#### 问题 4：简历驱动功能缺少统一前端入口
虽然架构文档已经加入简历能力，但前端需要明确新增：
- 简历上传入口
- 简历解析状态页
- 基于简历出题入口
- 基于简历模拟面试入口
- 简历画像与技术栈视图

## 3. 前端健壮性优化总原则

### 3.1 分页优先
凡是列表类数据展示，默认分页，不做全量加载。

适用页面：
- 题目列表页
- 学习记录页
- 模拟面试历史页
- 简历解析历史页
- 错题列表页
- 统计明细页

### 3.2 统一状态组件
所有页面统一处理：
- loading
- empty
- error
- partial data
- refreshing
- parsing
- generating
- submitting

### 3.3 AI 与确定性数据分层
前端展示时必须区分：
- 原始事实数据：题目标题、来源、时间、标签、知识节点
- AI 生成数据：答案、讲解、评分、追问、总结

### 3.4 任务化交互
对于上传、解析、出题、讲解、评分等异步流程，前端不能只给按钮，要显示：
- 当前任务状态
- 处理进度
- 失败原因
- 重试入口
- 结果链接

### 3.5 降级可用
后端或模型服务不稳定时，前端仍应支持：
- 浏览已有题库
- 查看已有学习记录
- 使用本地缓存或上次数据
- 跳过 AI 功能继续学习

## 4. 关键页面的前端交互优化策略

### 4.1 首页

#### 现状
首页已经有功能入口卡片、最近导入、快速统计。

#### 优化建议
- 增加“简历驱动面试”主入口
- 增加“最近简历”与“最近生成题目”卡片
- 增加“继续上次练习”快捷入口
- 增加任务通知区域，展示解析/出题/复习状态

#### 首页应只展示概览
首页不应拉取太多明细数据，只保留：
- 近期简历或题目
- 概览统计
- 快捷入口
- 最近任务

---

### 4.2 导入页

#### 现状
当前导入页已经支持文本导入和文件上传，交互基础较完整。

#### 优化建议
- 增加 Tab 或分区：
  - 面试题导入
  - 简历导入
- 简历导入后显示：
  - 解析状态
  - 提取到的技术栈
  - 提取到的项目经历
  - 是否可直接进入出题
- 导入完成后提供下一步动作按钮：
  - 查看解析结果
  - 直接生成题目
  - 进入模拟面试

#### 任务态建议
- 上传中
- 解析中
- 结构化中
- 已完成
- 解析失败可重试

#### 交互原则
简历导入不能只是“上传文件”，而必须是“上传后可立即看到解析价值”。

---

### 4.3 题目列表页

#### 现状
当前题目列表页有搜索和筛选，但仍然偏全量查询思路。

#### 优化建议
- 默认分页加载
- 筛选条件与分页联动
- 增加来源筛选：
  - 简历生成
  - 文件导入
  - 手动录入
  - AI 生成
- 增加知识域筛选
- 增加“仅看简历相关题目”快捷筛选
- 增加“最近新增”“最近练习”“错题”标签入口

#### 题目卡片建议补充信息
- 来源类型
- 是否来源于简历
- 关联技术栈
- 关联知识节点
- 是否已练习/已复习

#### 列表页分页策略
- 默认每页 20 条或 30 条
- 搜索和筛选均保留分页状态
- 页面切换后保持筛选条件
- 重置筛选时恢复第一页

---

### 4.4 题目详情页

#### 建议展示内容
- 题目正文
- 来源信息
- 简历证据片段
- 技术栈映射
- 关联知识节点
- 相关题目
- AI 讲解
- 用户历史练习记录
- 复习状态

#### 关键增强点
题目详情页是解释性最强的页面，建议突出“这道题为什么出现在这里”：
- 来自哪份简历
- 命中了哪段项目经历
- 关联哪些技术栈
- 为什么属于这个难度
- 为什么该题推荐追问

#### 视觉层级建议
把“来源链路”和“AI 讲解”分成两个明显区域，避免混在一起。

---

### 4.5 学习页

#### 现状
学习页已经是最成熟的业务页之一，具备练习、复习、统计三个模式。

#### 优化建议
- 练习模式增加“简历来源题目优先”开关
- 练习模式增加“按简历项目筛选”
- 复习模式增加来源分组：
  - 简历相关
  - 知识库相关
  - 错题相关
- 统计模式增加来源维度统计

#### 练习模式建议
- 支持随机抽题
- 支持按简历技术栈抽题
- 支持按项目经历抽题
- 支持按知识节点抽题
- 支持由浅入深出题策略

#### 复习模式建议
- 支持分页或分批加载待复习列表
- 支持按掌握度排序
- 支持显示上次复习结果和下一次复习时间

---

### 4.6 模拟面试页

#### 现状
已有领域和难度选择，但当前更像通用模拟面试，不够贴近个人简历。

#### 必须增强
- 增加“简历模式”
- 可选择简历中的某个项目
- 可选择简历中的某个技术栈
- 可选择整体模拟或局部深挖
- 可切换“通用题 / 简历题 / 混合题”模式

#### 建议的面试模式
1. **通用模式**：现有能力保留
2. **简历模式**：完全基于简历出题
3. **混合模式**：简历题 + 通用基础题
4. **深挖模式**：围绕一个项目持续追问

#### 面试流程建议
- 先选简历
- 再选项目或技术栈
- 再选择难度和模式
- 再开始面试
- 面试结束后输出简历画像与薄弱点

---

### 4.7 统计页

#### 现状
统计页已有基础指标与最近记录。

#### 优化建议
- 分页展示学习记录
- 可按来源类型过滤：简历 / 题库 / 错题
- 可按项目 / 技术栈过滤
- 可按练习类型过滤：练习 / 复习 / 面试
- 图表数据应尽量使用后端聚合接口，不拉原始全量记录

#### 统计页不应承担大数据列表渲染压力
统计页应尽量只拿：
- 聚合指标
- 最近 N 条记录
- 当前筛选下的轻量明细

## 5. 简历驱动功能的前端专项设计

### 5.1 简历入口设计
建议前端至少提供以下入口：

- 首页快捷入口“简历驱动面试”
- 导入页“简历导入”
- 模拟面试页“简历模式”
- 题目列表页“仅看简历相关题目”

### 5.2 简历解析结果展示
简历上传成功后，前端应展示：
- 简历文件名
- 解析状态
- 提取到的技术栈
- 提取到的项目经历
- 解析置信度
- 可生成的题目数量
- 下一步操作按钮

### 5.3 简历驱动出题展示
出题后要明确显示：
- 哪道题来源于简历哪一部分
- 属于什么技术栈
- 是基础题、项目题还是追问题
- 难度如何分布
- 是否已进入待练习列表

### 5.4 简历模式模拟面试
面试页要支持：
- 选择简历
- 选择项目经历
- 选择技术栈
- 选择难度
- 选择出题策略

---

## 6. 必须统一的前端状态设计

### 6.1 加载态
所有页面统一显示加载骨架或提示，不允许空白等待。

### 6.2 空态
当无数据时应提示：
- 原因
- 下一步动作
- 快捷入口

### 6.3 错误态
错误态要明确：
- 是网络问题
- 后端问题
- 模型问题
- 解析问题
- 权限问题

### 6.4 任务态
对于上传 / 解析 / 出题 / 评分 / 复习任务，要显示：
- 处理中
- 成功
- 失败
- 可重试
- 跳转查看结果

---

## 7. 数据加载策略

### 7.1 默认分页
所有展示列表默认分页，不做全量拉取。

### 7.2 首屏聚合优先
首页、统计页、概览页优先拿聚合数据。

### 7.3 详情页按需加载
题目详情、简历详情、会话详情只在进入时加载单条完整数据。

### 7.4 搜索防抖
搜索输入必须做防抖，避免每次键入都触发请求。

### 7.5 缓存与局部刷新
同一页面切换标签、筛选、分页时尽量保留缓存与局部更新。

## 8. 组件层统一建议

建议前端统一沉淀以下可复用组件：

- `PageHeader`
- `SectionCard`
- `EmptyState`
- `ErrorState`
- `LoadingState`
- `TaskStatusBadge`
- `SourceBadge`
- `AiGeneratedBadge`
- `DifficultyBadge`
- `KnowledgeNodeChip`
- `ResumeTechStackChip`
- `Pagination`
- `FilterBar`
- `ResultTimeline`

这样后续 Claude Code 增加页面时，不会重复造轮子。

## 9. 推荐的开发优先级

### 第一优先级
- 列表页分页
- 统一 loading / empty / error 状态
- 简历上传入口
- 简历模式模拟面试入口

### 第二优先级
- 题目详情来源链路展示
- 统计页聚合与分页
- 筛选器统一化
- 任务态与进度态

### 第三优先级
- 简历技术栈画像页
- 知识路径图
- 快捷入口与最近使用
- 降级策略与缓存策略

---

## 10. 给 Claude Code 的执行建议

当后续 Claude Code 依据本文件修改前端时，建议遵守以下顺序：

1. 先改数据接口使用方式，避免全量拉取
2. 再补通用分页组件与状态组件
3. 再补简历入口和简历模式入口
4. 再增强题目详情页和模拟面试页
5. 最后才做更复杂的统计和画像页面

不要一开始就追求“页面很炫”，优先追求“页面稳定、数据可控、功能可迭代”。
