# 09_Pagination_Module_Architecture

## 1. 目标

本项目的分页不是页面级临时实现，而是应该抽象成**可复用的通用分页体系**。其目标是：

- 让所有列表页分页逻辑保持一致
- 让前端页面只关心“展示什么”，不关心“如何分页”
- 让后端接口分页参数与返回结构统一
- 让搜索、筛选、排序、分页组合使用时行为一致
- 让未来增加新列表页时，能够直接复用现有分页模块

本文件定义分页模块的统一设计规范，适用于：
- 题目列表
- 简历列表
- 学习记录列表
- 错题列表
- 对话历史列表
- 复习任务列表
- 统计明细列表

---

## 2. 分页设计原则

### 2.1 统一原则

所有列表页必须遵循同一套分页规范，禁止每个页面单独发明分页参数、分页状态和分页 UI。

### 2.2 可复用原则

分页能力必须抽象成公用模块，而不是散落在每个页面内部。

### 2.3 可组合原则

分页要能与以下能力一起工作：
- 搜索
- 筛选
- 排序
- 刷新
- 缓存
- 导出
- 懒加载

### 2.4 一致性原则

所有页面在以下行为上必须一致：
- 默认每页大小
- 当前页切换方式
- 空数据展示
- 加载中状态
- 请求失败重试
- 筛选变更后如何重置页码

### 2.5 性能原则

- 默认分页，不允许全量拉取大列表
- 首页只展示概览，不展示全量数据
- 统计页只取聚合数据 + 当前页明细
- 详情页按需加载单条数据

## 3. 前端分页模块设计

### 3.1 目录建议

建议在前端抽象出统一分页模块：

```text
web/src/
├── components/
│   ├── pagination/
│   │   ├── Pagination.tsx
│   │   ├── PageSizeSelector.tsx
│   │   ├── PaginationSummary.tsx
│   │   └── index.ts
│   ├── states/
│   │   ├── LoadingState.tsx
│   │   ├── EmptyState.tsx
│   │   ├── ErrorState.tsx
│   │   └── index.ts
│   └── filters/
│       ├── FilterBar.tsx
│       └── index.ts
├── hooks/
│   ├── usePagination.ts
│   ├── usePaginatedQuery.ts
│   └── useDebouncedSearch.ts
└── lib/
    ├── api.ts
    └── pagination.ts
```

### 3.2 分页核心组件

#### `Pagination`
通用分页条，用于：
- 首页列表的少量数据分页
- 题目列表页
- 学习记录页
- 复习任务页
- 对话历史页

应支持：
- 首页 / 上一页 / 下一页 / 末页
- 当前页展示
- 总页数展示
- 跳页

#### `PageSizeSelector`
用于切换每页大小。默认建议提供：
- 10
- 20
- 30
- 50

#### `PaginationSummary`
显示当前页统计信息，例如：
- 第 21 - 40 条，共 287 条
- 当前页共 20 条结果

### 3.3 前端分页 Hook

#### `usePagination`
负责统一管理：
- `page`
- `pageSize`
- `offset`
- `limit`
- `total`
- `totalPages`
- `setPage`
- `setPageSize`
- `resetPage`

#### `usePaginatedQuery`
负责统一处理：
- 分页请求参数拼装
- 请求状态管理
- 筛选变化后重置页码
- 搜索防抖联动
- 缓存与局部刷新

#### `useDebouncedSearch`
负责搜索输入防抖，避免每次键入都触发分页请求。

## 4. 前端分页状态规范

所有分页页面统一使用以下状态字段：

| 状态字段 | 含义 |
|---|---|
| `page` | 当前页码 |
| `pageSize` | 每页条数 |
| `total` | 总条数 |
| `items` | 当前页数据 |
| `loading` | 是否加载中 |
| `error` | 是否错误 |
| `searchQuery` | 搜索关键词 |
| `filters` | 筛选条件 |
| `sort` | 排序条件 |

### 4.1 状态行为

- 搜索变化后：默认回到第一页
- 筛选变化后：默认回到第一页
- 页大小变化后：默认回到第一页
- 刷新后：保持当前筛选和页码，除非数据已失效

### 4.2 空态行为

如果当前页为空，页面必须判断是：
- 整体无数据
- 当前页超出范围
- 筛选条件过窄
- 搜索结果为空

并展示对应提示。

## 5. 后端分页接口规范

### 5.1 基础分页参数

所有列表 API 默认支持以下参数：
- `page`
- `page_size`
- 或 `offset` + `limit`

建议全项目统一一种风格，推荐优先使用：
- `page`
- `page_size`

因为更适合前端展示和跳页控制。

### 5.2 返回结构规范

建议统一返回：

```json
{
  "items": [],
  "total": 123,
  "page": 1,
  "page_size": 20,
  "total_pages": 7
}
```

### 5.3 后端统一分页响应类型

建议后端定义统一 Pydantic 泛型模型：
- `PaginatedResponse[T]`
- `PaginationMeta`

这样所有列表接口都采用同一返回格式。

### 5.4 搜索与分页

搜索接口必须支持分页，不允许一次返回全部搜索结果。

### 5.5 筛选与分页

筛选条件与分页必须联动，筛选变化后默认重置页码为 1。

## 6. 页面级统一分页策略

### 6.1 首页

首页只拿概览数据与最近少量数据：
- 最近 3 - 5 条题目
- 最近 3 - 5 条简历或任务
- 不做完整列表分页展示

### 6.2 题目列表页

- 默认分页
- 支持搜索分页
- 支持来源筛选分页
- 支持难度筛选分页
- 支持领域筛选分页

### 6.3 学习记录页

- 默认分页
- 支持按练习 / 复习 / 面试切换
- 支持按题目、简历、技术栈筛选

### 6.4 复习任务页

- 默认分页
- 优先展示待复习最紧急的数据
- 支持按掌握度排序

### 6.5 模拟面试历史页

- 默认分页
- 支持按简历、项目、领域筛选
- 支持查看单次会话完整详情

### 6.6 统计页

- 聚合接口优先
- 仅展示当前页明细
- 大规模数据不在前端全量渲染

---

## 7. 推荐的统一分页 UI

### 7.1 分页条样式要求

分页条应统一放置在列表底部，并包含：
- 总条数
- 当前页范围
- 分页按钮
- 每页数量选择
- 跳页输入（可选）

### 7.2 分页交互要求

- 当前页高亮
- 不可用按钮置灰
- 快速跳页时显示加载态
- 分页切换后保留筛选条件

### 7.3 移动端适配

移动端分页要简化为：
- 上一页
- 下一页
- 当前页/总页数

---

## 8. 建议的实现顺序

### 第一步：后端统一分页协议
先统一所有列表接口返回结构，确保前后端一致。

### 第二步：前端统一分页组件
再做一个公共分页组件和分页 Hook。

### 第三步：替换所有列表页
然后逐个替换：
- `questions`
- `study`
- `stats`
- 未来新增的简历页、历史页、复习页

### 第四步：统一缓存与状态组件
最后把 loading / empty / error / retry 与分页状态整合。

---

## 9. 给 Claude Code 的执行红线

当 Claude Code 修改任何列表页时，必须先检查：

- 该页面是否应该复用通用分页模块
- 该页面是否仍在全量请求
- 搜索和筛选是否与分页联动
- 后端接口是否返回统一分页结构
- 是否已经复用 `Pagination` / `usePaginatedQuery`

如果没有复用通用模块，说明实现不合格。

---

## 10. 最终结论

分页应该是一个公共基础设施，而不是每个页面自己实现一套。

本项目的目标不是“某一页能分页”，而是：
- **所有列表页的分页规范一致**
- **所有分页逻辑通过公用模块复用**
- **所有接口都能按统一协议分页**
- **所有新增页面都能直接接入现有分页体系**
