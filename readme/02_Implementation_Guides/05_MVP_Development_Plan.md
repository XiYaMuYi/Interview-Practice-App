# 05_MVP_Development_Plan

## 1. MVP 目标拆解

### MVP 1：最小可用版
目标是先跑通“导入资料 → 自动整理 → 可浏览可检索”的闭环。

核心能力：
- 文本粘贴
- 文件上传
- 简历上传与结构化解析
- 题目识别
- 自动分类
- 标签管理
- 列表展示
- 搜索筛选
- 题目详情页
- 根据简历生成强相关面试题

### MVP 2：智能学习版
目标是在 MVP 1 基础上增加“讲解、排序、复习、对话”的能力。

核心能力：
- 难度排序
- 学习路径推荐
- 答案分层生成
- 错题集
- 学习记录
- 模拟问答
- LangGraph 智能体基础流程

### 1.1 架构演进基调

本项目采用 **模块化单体** 路线。MVP 的任务优先级不是“最快把功能堆出来”，而是“先把边界定好，再逐步扩展”。

### 1.2 新增 MVP 核心能力：简历驱动出题

简历上传与结构化解析必须作为 MVP 核心能力之一。系统必须根据简历里的技术栈、项目经历、职责描述、成果描述来生成面试题，而不是凭空出题。出题必须能够追溯到简历原文，确保每一道题都“有来源、可解释、可复盘”。

#### 必须遵守的实现顺序
1. 先搭 `LLMGateway` 与 `PromptRegistry`
2. 再搭 `services/` 业务编排层
3. 再写 `infra/` 的数据库、文件、解析、向量适配
4. 再写 `api/` 路由
5. 最后接入前端页面

#### 强制约束
- API 不能直接调大模型
- API 不能直接写数据库查询
- 前端不能依赖未稳定的内部模型结构
- 所有 AI 调用必须先经过统一网关
- 核心表必须预留扩展字段，不能因 MVP 简化掉

## 3. 具体任务清单

### 3.1 后端任务

#### 基础工程
- 初始化 FastAPI 项目结构
- 配置环境变量与分层目录
- 配置 PostgreSQL 与 Alembic
- 配置 pgvector
- 设计基础响应模型与异常处理
- 初始化统一 `LLMGateway` 与 `PromptRegistry`
- 建立简历解析、简历映射、简历出题三个服务边界
- 建立统一任务化与流式推送基础设施，支持长任务增量保存与进度回传
- 明确 LangGraph 的使用边界：普通抽取/分类/讲解/复习使用线性 StateGraph，动态追问、动态出题、基于上下文循环决策的面试官 Agent 使用 ReAct 范式
- 把 ReAct 作为关键场景能力而不是全局默认：只用于需要“思考→工具调用→观察→再思考”的任务

#### 文件导入
- 实现文件上传接口
- 实现粘贴文本导入接口
- 实现文件元信息保存
- 实现解析状态更新
- 预留同步解析与异步解析接口
- 增加简历上传专用接口与解析流程
- 所有长文档导入改为任务化 + 流式进度推送 + 增量保存

#### 文档解析
- PDF 解析
- Word 解析
- 图片 OCR 解析
- 文本清洗与段落切分

#### 题目与标签
- 题目入库接口
- 自动标签生成接口
- 难度评估接口
- 题目查询与过滤接口
- 题目知识节点关联接口
- 简历技术栈到题目标签的映射接口
- 简历经历到题目生成的接口

#### 智能体与对话
- LangGraph 工作流最小实现
- 题目讲解接口
- 模拟面试接口
- 评分与复盘接口
- 简历驱动出题工作流节点
- 简历相关追问工作流节点
- 讲解、出题、追问等长生成流程支持流式输出与增量落库
- 面试官 Agent 必须支持 ReAct 循环：根据简历、题库、知识库、用户回答动态选择工具与下一步动作
- 不是所有 LangGraph 流都做 ReAct，只有需要循环决策和工具调用的场景才使用
- 所有 ReAct / 循环型 Agent 必须设置 `max_turns` / `max_iterations` / `max_steps` 上限，防止无限循环和额外费用消耗
- 到达上限后必须强制结束，并输出可解释的收敛结果或阶段性总结

#### 复习系统
- 错题记录接口
- 学习记录接口
- 下次复习时间计算逻辑
- 简化 SM-2 策略实现

#### 认证与权限
- 登录接口基础实现
- JWT / 认证中间件
- 配置开关控制公开访问
- 默认隐藏前端登录入口

## 6.1 前端交互设计（同步版）

> 这一部分是给 Claude Code 直接执行的前端交互设计清单。要求在现有前端基础上做微调，而不是推倒重做。

### 6.1.1 设计目标

- 让用户可以上传简历并进入“简历驱动出题”流程
- 让用户可以从简历生成的题目直接进入模拟面试
- 让题目、简历证据、追问、评分、复盘形成闭环
- 让现有题库、学习练习、统计页面继续兼容简历入口

### 6.1.2 页面与入口调整

#### 首页 `/`
- 新增一个主入口卡片：`简历驱动面试`
- 新增入口描述：上传简历，自动生成基于项目经历和技术栈的面试题
- 首页快速入口保持原有功能不变

#### 导入页 `/import`
- 保留原有“文本导入”“文件上传”
- 新增第三个区域或 Tab：`简历导入`
- 简历导入支持拖拽上传、粘贴文本、查看解析状态
- 简历导入后展示：技术栈摘要、项目经历摘要、可出题方向、下一步按钮

#### 题目列表页 `/questions`
- 新增筛选维度：`来源类型`
- 支持筛选 `简历生成`、`手动导入`、`文本导入`、`文件导入`
- 每条题目卡片增加来源标识：`来自简历 / 来自题库 / 来自文本`
- 支持查看题目是否关联了简历证据或知识节点

#### 题目详情页 `/questions/[id]`
- 增加“来源证据”区域
- 若题目来自简历，显示对应项目经历或技术栈片段
- 增加“基于该简历出题”提示
- 增加“开始模拟面试”“继续追问”“重新生成相似题”按钮

#### 模拟面试页 `/interview`
- 增加“简历模式”开关
- 简历模式下，用户先选择某份简历或某个项目经历
- 系统根据简历内容拉起第一题，而不是手工选择领域
- 面试过程显示当前题目来源、关联项目、关联技术点、难度等级
- 面试结束后显示与该简历相关的薄弱点总结

#### 学习页 `/study`
- 新增“按简历复习”入口
- 支持从简历派生题目中选择错题集、薄弱项、重点追问项

#### 统计页 `/stats`
- 新增简历训练统计：
  - 简历导入次数
  - 简历派生题目数量
  - 简历相关练习轮次
  - 薄弱技术栈分布
  - 项目追问命中率

### 6.1.3 关键交互流

#### 流程 A：简历上传到出题
1. 用户进入导入页
2. 选择“简历导入”
3. 上传简历文件或粘贴简历文本
4. 前端展示解析进度
5. 后端返回结构化简历结果
6. 前端展示技术栈、项目经历、可出题方向
7. 用户点击“开始基于简历练习”
8. 跳转到模拟面试页并自动载入第一题

#### 流程 B：简历题进入追问
1. 用户在模拟面试页回答简历相关问题
2. 后端根据简历证据生成追问
3. 前端展示追问链路和评分结果
4. 用户可继续回答或直接查看总结

#### 流程 C：从题目详情进入简历关联学习
1. 用户在题目详情页看到来源证据
2. 点击关联项目或技术栈
3. 前端跳转到同类题推荐或模拟面试页
4. 系统继续基于简历上下文出题

### 6.1.4 前端实现原则

- 不改掉现有页面结构，只做增量增强
- 新功能入口优先用卡片、Tab、筛选器、按钮补入
- 简历模式与普通题库模式必须共存
- 前端只消费稳定 DTO，不直接依赖后端内部字段结构
- 所有新增交互都必须兼容未来多用户与权限开放
- 登录入口仍然默认隐藏，但路由和弹层可以先保留

## 9. 风险与应对

### 风险 1：文件解析质量不稳定
应对：优先支持文本粘贴，文件解析作为增强能力。

### 风险 2：AI 输出不稳定
应对：统一结构化输出，所有结果写入 schema。

### 风险 3：功能过多导致延期
应对：先做 MVP 1，再逐步加智能学习与对话。

### 风险 4：页面复杂度上升
应对：前端采用轻量统一组件库，先保证信息清晰。
